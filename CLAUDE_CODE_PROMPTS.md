# HARstack — Claude Code handoff prompts

A sequenced set of prompts to migrate this baseline into a fresh GitHub project,
verify the local build, and productize the extension. Run them in order, in a
Claude Code session opened at the root of the new repo. Each prompt is
self-contained; paste one, let it finish, then paste the next.

The non-prompt setup steps you do yourself (create the repo, get API keys,
configure Pages) are called out as **[YOU]** between prompts.

---

## PROMPT 0 — Orientation (paste first, every session)

```
You are working on HARstack, a privacy and martech stack auditor. Read README.md
and SETUP.md before doing anything, then confirm the layout back to me in three
lines.

Hard design constraints you must never violate:
- The browser extension is a DevTools panel. It must request ZERO permissions
  (an empty `permissions` array). Never add host permissions, webRequest, debugger,
  declarativeNetRequest, storage, or a background service worker. The whole product
  promise is zero permissions.
- `extension/engine.js` is GENERATED by build/build.py. Never hand-edit it.
  The engine source is build/source/harstack-source.html; the tracker registry
  is build/trackers.yaml. Changes to detection go there, then rebuild.
- Writing voice for all docs and site copy: short sentences, active voice, no em
  dashes, no exclamation points, no marketing fluff. Plain and direct.

Do not change application logic in this session unless I ask. Acknowledge these
constraints before proceeding.
```

**[YOU]** Create the new empty repo on GitHub first (e.g. `harstack` under your
account). Clone it locally. Unzip this baseline into it so the files sit at the
repo root. Then open Claude Code there.

## PROMPT 1 — Verify the local build

```
Set up the local build and prove it works.

1. Confirm Python 3.9+ and Node 18+ are available; print versions.
2. Create a virtualenv at .venv, install pyyaml into it, and document that in
   SETUP.md if it is not already there.
3. Run the build:  cd build && python build.py
4. Confirm all four outputs were (re)generated: harstack.html, tool/index.html,
   extension/engine.js, and that "sanity_check_both_outputs: ALL CHECKS PASSED"
   printed.
5. Confirm the generated extension engine is the real engine, not a stub:
   grep for "__isStub: false" in extension/engine.js and confirm there is no
   leftover "// INJECT:TRACKERS" placeholder.
6. Run the extension regression test:  node extension/tests/engine.test.js
   All checks must pass.

Report a short pass/fail table. Do not modify build.py unless a step fails; if it
fails, show me the error before changing anything.
```

## PROMPT 2 — Build ergonomics and the extension load flow

```
Add a small build/dev harness without changing application logic.

1. Create a Makefile at repo root with targets:
   - `build`   -> cd build && python build.py
   - `test`    -> node extension/tests/engine.test.js  (and build/tests/validate.js
                  if fixtures exist; skip cleanly if the fixtures/ dir is absent)
   - `ext-chrome`  -> copy extension/manifest.chrome.json to extension/manifest.json
   - `ext-firefox` -> copy extension/manifest.firefox.json to extension/manifest.json
   - `lint`    -> npx web-ext lint --source-dir=extension  (note in output if web-ext
                  is not installed, with the install command)
   - `clean`   -> remove generated artifacts (extension/engine.js, and optionally the
                  built html if I confirm)
2. Add a package.json with a "scripts" block mirroring those, plus web-ext as a
   devDependency. Do not add any runtime dependencies.
3. Update README.md "Build the extension" section to reference `make build`,
   `make ext-chrome`, and the load-unpacked steps. Keep the existing voice.

After this, run `make build && make test` and show me the result.
```

## PROMPT 3 — Productize: the extension landing page and site wiring

```
Wire the extension landing page into the GitHub Pages site and tighten the copy.

Context: index.html (repo root) is the live harstack.com landing page. The
extension landing page already exists at extension/site/index.html and reuses the
same design tokens (Libre Baskerville / IBM Plex Mono / DM Sans, ink/paper/accent
palette). The site is served from the repo root via GitHub Pages, custom domain
harstack.com (see CNAME).

Tasks:
1. Decide the public URL for the extension page. Recommend /extension/ as the path.
   Because the extension/ folder also holds source, copy the landing page to the
   served location during build instead of serving the source folder directly:
   add a step to build/build.py that copies extension/site/index.html to a
   published path. Propose the cleanest option (e.g. a top-level /plugin/ directory
   that Pages serves, fed from extension/site/) and implement it. Do not expose the
   raw extension source files (engine.js, panel.js) at a public URL.
2. Add a nav link to the extension page from the main index.html, in the existing
   dateline/nav style. Add a short "Now available as a browser extension" callout
   in an appropriate section of the main page, matching the existing voice.
3. Update sitemap.xml and llms.txt to include the new page.
4. Verify all internal links resolve and the page renders. Show me a screenshot or
   the rendered structure.

Keep the writing voice: short sentences, active voice, no em dashes, no fluff.
```

**[YOU]** In the new repo's GitHub settings, enable Pages (Deploy from branch:
`main`, root). Confirm the `CNAME` file still reads `harstack.com` and that your
DNS still points at GitHub Pages. The custom domain carries over only if DNS is
intact.

## PROMPT 4 — Distribution and release

```
Set up reproducible releases for both browsers without adding permissions.

1. Write a release script (scripts/release.sh) that:
   - runs `python build/build.py`
   - runs the regression test and aborts on failure
   - produces two zips: harstack-extension-chrome.zip (with manifest.chrome.json
     copied to manifest.json) and harstack-extension-firefox.zip (firefox manifest),
     each containing the extension/ folder WITHOUT the two source manifests, the
     site/ folder, or the tests/ folder. Include the generated engine.js.
   - stamps the version: read it from a single source (propose extension/manifest.*.json
     or a VERSION file) and confirm engine + manifests agree.
2. Document the Firefox signing path in SETUP.md: `web-ext sign --channel=unlisted`
   needs an AMO API key/secret and the gecko id already set in manifest.firefox.json.
   Note that release Firefox will not install unsigned; signed unlisted xpi installs
   permanently.
3. Document the Chrome path: load unpacked for personal use; Web Store listing for
   public distribution. Note that off-store .crx will not install in normal Chrome.
4. Add a GitHub Actions workflow (.github/workflows/build.yml) that runs the build
   and the regression test on push and PR. Do NOT auto-publish; build and test only.

Show me the release script and the workflow. Do not commit any API secrets.
```

**[YOU]** For Firefox signing: log in at addons.mozilla.org, Developer Hub →
Manage API Keys, generate the JWT issuer and secret, and keep them in your local
environment, never in the repo.
